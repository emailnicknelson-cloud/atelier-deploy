The Problem We're Solving

The $150B Content Bottleneck

3D asset creation is the single most expensive, slowest, and most exclusionary bottleneck in the digital content supply chain. The problem manifests across three distinct user segments — each one a multi-billion dollar market locked behind the same gate:

The Solution

What 3DAgent Actually Is

3DAgent is a multi-agent orchestration pipeline where 10+ specialized AI agents collaboratively produce production-quality Blender files. It's not a model — it's a factory.

Prompt → Director → Modeler → [QA Gate] → UV Mapper → Optimizer → Rigger → Material Artist → Lighter → Compositor → Renderer → [QA] → Export
LLM Agent Local Script (0 tokens) Quality Gate

Three Architectural Choices That Make This Different

1

Multi-Agent Specialization Over Monolithic Generation

Each agent has domain-specific Blender API knowledge baked into its system instructions. The Modeler knows bmesh topology patterns. The Material Artist knows PBR-accurate roughness values. The Lighter knows 3-point studio setups. No single LLM call can orchestrate all of this — the pipeline itself is the product.

2

Deterministic Safety Nets at Zero Token Cost

Five local scripts (UV, Optimizer, Material Clamp, Tech Artist, Export) run algorithmically — no LLM, no tokens. They guarantee baseline quality even when AI agents fail. Materials will always be PBR-correct. UVs will always be unwrapped. Geometry will always be cleaned.

3

Self-Correcting Execution with Persistent Memory

A 30+ rule sanitizer auto-fixes known Blender API mistakes (training data is full of 2.x/3.x patterns that crash in 4.0). A correction loop feeds tracebacks back to the LLM. ChromaDB stores every successful script for semantic retrieval on future runs. The system literally gets better at making things the more you use it. A first-attempt success rate of ~30% becomes a 95%+ pipeline completion rate.

The Opportunity Space

What Can We Uniquely Do That Nobody Else Can?

1

Design Reasoning Before Execution

The Director agent analyzes the prompt and creates a structured execution plan — decomposing "medieval blacksmith forge" into constituent meshes, materials, and spatial relationships. This isn't random generation; it's architectural thinking.

2

Multi-File, Multi-Step Orchestration

Each agent reads the current state of the .blend file (via introspection), writes a targeted Python script, and hands off to the next agent. The file evolves through 10+ transformation steps. No single-shot generator can do this.

3

Persistent Learning

ChromaDB semantic retrieval means the system surfaces the best past scripts for similar prompts. The 112th barrel benefits from the 1st barrel's lessons. This is compound intelligence, not stateless inference.

4

Quality Gates as First-Class Citizens

Geometry validation (vertex count, degenerate faces, volume checks) and visual quality scoring (Gemini vision compares render vs. prompt) happen inside the pipeline, not after. Bad geometry gets caught before materials are applied.

Constraints & Risks

What We Can't Do (Yet)

LimitationWhyMitigation
Complex organic geometry LLMs script with primitives + modifiers, not sculpted meshes. A dragon will be recognizable but won't match hand-sculpted quality. Position as props/environment, not hero characters.
Mesh-level surface detail No sculpt brushes, no retopology. Detail comes from procedural textures, not geometry. Lean into material quality — procedural textures carry significant visual weight at game-engine distances.
Characters & animation Rigging exists but character topology needs human-level understanding of edge flow. Out of scope for v1. Revisit with mesh generation models later.
Deterministic output Same prompt → different geometry each time. Actually a feature for batch generation (variety), but a risk for "exact match" use cases.

Key Risks

!

Gemini API Changes / Pricing Shifts

Already structured to swap models (fallback chain: gemini-2.5-pro → 2.5-flash → 2.0-flash). Local scripts mean 50% of the pipeline works with zero API.

!

Mesh Generation Models Get Good Enough

If Meshy/Rodin start outputting production-ready .blend files, our orchestration layer becomes less valuable. Counter: they won't have our pipeline finishing, quality gates, or persistent memory.

!

Blender Version Drift

Training data lags real API. Counter: our sanitizer already handles 30+ known 2.x→4.0 breakages, and the safe API abstracts the most fragile calls.
